ACTIVE 幕一：公开仓库 >

事件最早并不是从主机告警开始的。
它是从一份 "过于干净" 的公开仓库开始的。
如果公开材料不完整，那么最值得看的，通常不是它留下了什么，而是它试图删掉什么。

  • clone 仓库
  • 运行基础分析命令
  • 检查日志与旧文件
  • 查看 git 历史中是否存在 internal-only 配置痕迹
  • 确认是否能将样本指向某台具体主机
打开仓库
你已经确认：公开仓库并非最终版本。
一份 responder-only 索引将事件重新指向了一台具体主机。
下一步，调查对象不再是 "仓库"，而是 "主机上发生过什么"。

Check old commits before trusting current files.

ACTIVE 幕二：受害主机源码残片 >

受害主机已下线。
信息中心从 Web 根目录、部署目录与日志目录中恢复出部分 Java / JSP 源码残片。
它们不完整，但足够说明一件事：这台主机曾经替别人执行过命令。

查看源码残片 查看访问日志 查看进程树

而现在要追的，不再是访问了哪一页，而是那一页后来替谁下载了什么。

The first obvious shell is not always the real shell.

ACTIVE 幕三：主机侧异常与矿工落地 >

命令执行只是起点。
在主机侧，攻击者没有着急带走数据，也没有制造显眼的破坏。
他们只留下一个更安静的目标：让这台机器长期替他们工作。

  • 检查 shell 执行后的落地文件
  • 查看 cron / systemd / 历史命令
  • 判断恶意程序如何持久化
  • 从配置和日志中确认矿池、钱包、worker
查看主机痕迹 查看矿工配置 查看网络连接
已确认：这不是一次数据窃取事件。
这是一次资源劫持。
你已经找到收益流向。

A quiet host with high CPU is still talking.
